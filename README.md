# Интернет-магазин «Web-Larёk»

«Web-Larёk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер.

## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component

Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент, за отображение которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`

#### Класс Api

Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter

Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` - хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.

### Данные

#### Интерфейс IProduct - Товар

Поля интерфейса:\
`id: string` - ID товара;\
`description: string` - описание товара; \
`image: string` - ссылка на изображение товара;\
`title: string` - название товара;\
`category: string` - категория товара;\
`price: number | null` - цена товара.

#### Интерфейс IBuyer - Покупатель

Поля интерфейса:\
`payment: TPayment` - выбранный тип оплаты;\
`email: string` - email покупателя;\
`phone: string` - номер телефона покупателя;\
`address: string` - адрес доставки.

#### Интерфейс IOrder

Описывает объект, передаваемый на сервер при размещении заказа. Наследует интерфейс IBuyer.

Дополнительные поля интерфейса:\
`total: number` - итоговая сумма заказа;\
`items: string[]` - список заказанных товаров.

#### Интерфейс IOrderResponse

Описывает объект, получаемый от сервера в ответ на запрос - размещение заказа.

Поля интерфейса:\
`id: string` - ID заказа;\
`total: number` - сумма списания.

#### Интерфейс IProductsResponse

Описывает объект со списком товаров, получаемый от сервера.

Поля интерфейса: \
`items: IProduct[]` - список товаров;\
`total: number` - количество полученных записей товаров.

### Модели данных

#### Класс Catalogue

Хранит каталог товаров, доступных к покупке в web-ларьке.

Поля класса:\
`private _items: IProduct[] = []` - массив всех товаров;\
`private _itemActive: IProduct | null = null` - товар, выбранный для подробного отображенния.

Конструктор принимает брокер событий `events: IEvents` в качестве параметра.

Методы класса:\
`set items(items: IProduct[])` - сеттер для сохранения массива товаров и вызывает событие изменения каталога; \
`get items(): IProduct[]` - геттер для получения массива товаров; \
`set itemActive(item: IProduct)` - сеттер для сохранения товара для подробного отображения; \
`get itemActive(): IProduct | null` - геттер для получения товара для подробного отображения и вызывает событие установки активного товара; \
`getItem(id: string): IProduct | undefined` - получение одного товара по его ID.

#### Класс Cart

Хранит список товаров, выбранных покупателем для заказа.

Поля класса:\
`private _items: IProduct[]` - массив товаров, выбранных покупателем.

Конструктор принимает брокер событий `events: IEvents` в качестве параметра.

Методы класса:\
`get items(): IProduct[]` - получение массива товаров, которые находятся в корзине; \
`add(item: IProduct): void` - добавление товара, который был получен в параметре, в массив корзины и вызывает событие изменения корзины; \
`delete(item: IProduct): void` - удаление товара, полученного в параметре из массива корзины и вызывает событие изменения корзины; \
`clear(): void` - очистка корзины и вызывает событие изменения корзины; \
`total(): number` - получение стоимости всех товаров в корзине; \
`count(): number` - получение количества товаров в корзине; \
`checkItem(id: string): boolean` - проверка наличия товара в корзине по его id, полученного в параметр метода.

#### Класс Buyer

Хранит данные покупателя, вводимые при оформлении заказа.

Поля класса:\
`payment: TPayment` - выбранный тип оплаты;\
`email: string` - email покупателя;\
`phone: string` - номер телефона покупателя;\
`address: string` - адрес доставки.

Конструктор принимает брокер событий `events: IEvents` в качестве параметра.

Методы класса: \
`set data(data: Partial<IBuyer>)` - сохранение всех полученных данных, вызов события обновления данных покупателя; \
`get data(): IBuyer` - получение всех данных покупателя; \
`clear(): void` - очистка данных покупателя, вызов события обновления данных покупателя; \
`validate(): TErrors` - валидация полученных данных.

Поля класса:\
`private _baseApi: IApi` - объект с настройками класса Api, используемого через композицию.

Методы класса:\

`item-selected` - изменение выбранного для просмотра товара;\
`cart-changed` - изменение содержимого корзины;\
`buyer-updated` - изменение данных покупателя.

### Презентер

В проекте используется паттерн MVP с брокером событий.

Сохранение в модель списка товаров каталога, полученного с сервера, вызывает событие `catalogue-ready`, в результате обработки которого происходит отрисовка каталога товаров.

Кнопка - карточка товара вызывает событие `choose-card`, записывающее в модель данных выбранный товар. Это, в свою очередь, вызывает событие `item-selected`, приводящее к отрисовке подробной карточки товара.

Событие `cart-action` вызывается с кнопки "Добавить в корзину" / "Удалить из корзины" в подробной карточке товара, а также с кнопки удаления товара из корзины в модальном окне корзины. В результате его работы в модель корзины передается товар, который необходимо удалить или добавить. Событие одно для добавления и удаления из корзины, поскольку текст кнопки "Добавить"/"Удалить" может быть получен только из модели, а следовательно отражает действительное наличие или отсутствие товара в корзине.

При внесении изменений в модель корзины вызывается событие `cart-changed`: перерисовывается блок с корзиной и меняется текст кнопки в подробной карточке товара.

Нажатие кнопки с иконкой корзины `basket:open` приводит к открытию модального окна с корзиной. В метод `.render()` не передаются данные, поскольку при первичной загрузке страницы корзина пуста, а при любом изменении корзины сразу происходит перерисовка списка товаров в ней. Таким образом, при любом открытии корзины список товаров в ней актуален.

Событие `modal:close` вызывается с кнопки "За новыми покупками!" и закрывает модальное окно. В остальных случаях закрытие модального окна возможно изнутри класса отображения модального окна. 

Событие `buy-cart` происходит при нажатии кнопки "Оформить" в корзине и вызывает переход к первому шагу оформления заказа с получением данных покупателя из модели.

По кнопке "Далее" в первой форме вызывается событие `submit-order`, которое отрисовывает форму второго шага оформления заказа с получением данных покупателя из модели.

Событие `form-update` вызывается любым изменением полей в формах и передает данные форм в модель.

Событие `buyer-updated` реагирует на изменение данных покупателя в модели, запускает валидацию форм и передает в отображение тексты ошибок.

Событие `submit-contacts` вызывается нажатием кнопки "Оплатить" во втором шаге формы, отправляет данные заказа на сервер и с полученным ответом отрисовывает модальное окно успешной покупки с подстановкой списанной суммы.
